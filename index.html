Singleton Pattern

Creational Patterns
import React from 'react';

const Singleton = (() => {
  let instance;

  const createInstance = () => {
    const object = new Object("I am the singleton instance");
    return object;
  };

  return {
    getInstance: () => {
      if (!instance) {
        instance = createInstance();
      }
      return instance;
    }
  };
})();

const getSingletonInstance = () => {
  return Singleton.getInstance();
};

const App = () => {
  const instance = getSingletonInstance();

  return (
    <div>
      <h1>Singleton Pattern Example</h1>
      <p>{instance.toString()}</p>
    </div>
  );
};

export default App;

Pros

  • Global Access: Provides a single point of access to a shared resource across the application.
  • Lazy Initialization: The instance is created only when it's first requested, saving resources.
  • Thread Safety: When implemented correctly, ensures only one instance exists even in multi-threaded environments.
  • Consistency: Guarantees that all objects use the same instance throughout the application.

Cons

  • Global State: Can make the application's behavior less predictable and harder to debug.
  • Tight Coupling: Classes that depend on the Singleton become tightly coupled to it.
  • Testability: Can make unit testing more difficult due to its global state.
  • Violation of Single Responsibility Principle: The Singleton class takes on the responsibility of managing its own lifecycle.

Factory Pattern

Creational Patterns
import React from 'react';

const componentFactory = (type) => {
  switch (type) {
    case 'Button':
      return <Button />;
    case 'Input':
      return <Input />;
    case 'Checkbox':
      return <Checkbox />;
    default:
      return null;
  }
};

const Button = () => {
  return <button>Click Me</button>;
};

const Input = () => {
  return <input type="text" placeholder="Enter text" />;
};

const Checkbox = () => {
  return <input type="checkbox" />;
};

const App = () => {
  const [componentType, setComponentType] = React.useState('Button');

  return (
    <div>
      <h1>Factory Pattern Example</h1>
      <select onChange={(e) => setComponentType(e.target.value)} value={componentType}>
        <option value="Button">Button</option>
        <option value="Input">Input</option>
        <option value="Checkbox">Checkbox</option>
      </select>
      <div>
        {componentFactory(componentType)}
      </div>
    </div>
  );
};

export default App;

Pros

  • Abstraction: Encapsulates object creation logic, allowing the client code to focus on using the object rather than creating it.
  • Flexibility: Makes it easy to introduce new types of components without changing existing code.
  • Centralized Control: Provides a centralized point for object creation, making it easier to manage and modify the creation process.
  • Loose Coupling: Reduces dependencies between the client code and the concrete classes being instantiated.

Cons

  • Complexity: Can introduce unnecessary complexity for simple object creation scenarios.
  • Increased Number of Classes: May lead to a proliferation of factory classes, especially if there are many different types of objects.
  • Harder to Subclass: If the factory method is in a base class, all subclasses must override it, which can be cumbersome.
  • Performance Overhead: In some cases, using a factory method might introduce a slight performance overhead compared to direct object instantiation.

Abstract Factory

Creational Patterns
import React from 'react';

const createUIFactory = (type) => {
  switch (type) {
    case 'MaterialUI':
      return materialUIFactory;
    case 'BootstrapUI':
      return bootstrapUIFactory;
    default:
      throw new Error('Unknown UI type');
  }
};

const materialUIFactory = {
  createButton: () => <MaterialButton />,
  createInput: () => <MaterialInput />,
};

const bootstrapUIFactory = {
  createButton: () => <BootstrapButton />,
  createInput: () => <BootstrapInput />,
};

const MaterialButton = () => {
  return <button style={{ backgroundColor: 'blue', color: 'white' }}>Material Button</button>;
};

const MaterialInput = () => {
  return <input type="text" placeholder="Material Input" style={{ borderColor: 'blue' }} />;
};

const BootstrapButton = () => {
  return <button className="btn btn-primary">Bootstrap Button</button>;
};

const BootstrapInput = () => {
  return <input type="text" className="form-control" placeholder="Bootstrap Input" />;
};

const App = () => {
  const [uiType, setUIType] = React.useState('MaterialUI');
  const uiFactory = createUIFactory(uiType);

  return (
    <div>
      <h1>Abstract Factory Pattern Example</h1>
      <select onChange={(e) => setUIType(e.target.value)} value={uiType}>
        <option value="MaterialUI">Material UI</option>
        <option value="BootstrapUI">Bootstrap UI</option>
      </select>
      <div>
        {uiFactory.createButton()}
        {uiFactory.createInput()}
      </div>
    </div>
  );
};

export default App;

Pros

  • Flexibility: Allows creating families of related objects without specifying their concrete classes.
  • Consistency: Ensures that products from the same family are used together.
  • Separation of Concerns: Isolates the client code from the details of product creation.
  • Easy Product Family Switch: Makes it simple to switch entire product families.

Cons

  • Complexity: Can be overkill for simple scenarios, adding unnecessary abstraction.
  • Difficulty in Adding New Products: Introducing new types of products can require changes to the abstract factory interface and all its implementations.
  • Increased Number of Classes: Results in more interfaces and classes, which can make the codebase harder to navigate.
  • Rigid Structure: Once established, the structure can be difficult to modify without affecting multiple parts of the system.

Builder Pattern

Creational Patterns
import React from 'react';

const createFormBuilder = () => {
  const form = {
    title: '',
    fields: [],
    submitButton: null,
  };

  return {
    setTitle: (title) => {
      form.title = title;
      return this;
    },
    addField: (field) => {
      form.fields.push(field);
      return this;
    },
    setSubmitButton: (button) => {
      form.submitButton = button;
      return this;
    },
    build: () => form,
  };
};

const createTextField = (label, placeholder) => ({
  type: 'text',
  label,
  placeholder,
});

const createSubmitButton = (label) => ({
  type: 'submit',
  label,
});

const Form = ({ form }) => (
  <form>
    <h2>{form.title}</h2>
    {form.fields.map((field, index) => (
      <div key={index}>
        <label>{field.label}</label>
        <input type={field.type} placeholder={field.placeholder} />
      </div>
    ))}
    {form.submitButton && (
      <button type={form.submitButton.type}>{form.submitButton.label}</button>
    )}
  </form>
);

const App = () => {
  const formBuilder = createFormBuilder()
    .setTitle('Contact Us')
    .addField(createTextField('Name', 'Enter your name'))
    .addField(createTextField('Email', 'Enter your email'))
    .setSubmitButton(createSubmitButton('Submit'))
    .build();

  return (
    <div>
      <h1>Builder Pattern Example</h1>
      <Form form={formBuilder} />
    </div>
  );
};

export default App;

Pros

  • Flexibility: Allows step-by-step construction of complex objects.
  • Separation of Concerns: Separates the construction of a complex object from its representation.
  • Control over Construction Process: Provides more control over the steps of construction.
  • Reusability: Can reuse the same construction process to create different representations.

Cons

  • Complexity: Can lead to more complex code, especially for simpler objects.
  • Increased Number of Classes: Requires creating multiple new classes for each type of object.
  • Mutable Objects: The object being built is typically mutable during construction.
  • Performance Overhead: May introduce slight performance overhead due to the additional abstraction layer.

Prototype Pattern

Creational Patterns
import React from 'react';

const formPrototype = {
  title: '',
  fields: [],
  submitButton: null,
  clone() {
    return { ...this, fields: [...this.fields] };
  },
};

const createForm = (title, fields, submitButton) => {
  const form = formPrototype.clone();
  form.title = title;
  form.fields = fields;
  form.submitButton = submitButton;
  return form;
};

const Form = ({ form }) => (
  <form>
    <h2>{form.title}</h2>
    {form.fields.map((field, index) => (
      <div key={index}>
        <label>{field.label}</label>
        <input type={field.type} placeholder={field.placeholder} />
      </div>
    ))}
    {form.submitButton && (
      <button type={form.submitButton.type}>{form.submitButton.label}</button>
    )}
  </form>
);

const App = () => {
  const form1 = createForm(
    'Contact Us',
    [
      { type: 'text', label: 'Name', placeholder: 'Enter your name' },
      { type: 'text', label: 'Email', placeholder: 'Enter your email' },
    ],
    { type: 'submit', label: 'Submit' }
  );

  const form2 = createForm(
    'Feedback',
    [
      { type: 'text', label: 'Comments', placeholder: 'Enter your comments' },
    ],
    { type: 'submit', label: 'Send Feedback' }
  );

  return (
    <div>
      <h1>Prototype Pattern Example</h1>
      <Form form={form1} />
      <Form form={form2} />
    </div>
  );
};

export default App;

Pros

  • Simplified Object Creation: Allows creating new objects by copying an existing instance, reducing the need for complex initialization code.
  • Flexibility: Enables adding or removing products at runtime by cloning existing prototypes.
  • Reduced Subclassing: Can be used as an alternative to inheritance for varying product configurations.
  • Dynamic Loading: Allows for adding classes to the system without modifying it.

Cons

  • Complexity with Circular References: Cloning objects with circular references might be challenging and lead to complex code.
  • Deep Copy vs Shallow Copy: Implementing a deep copy for complex objects can be tricky and potentially impact performance.
  • Overuse: Not every object needs to be cloneable, and overuse can lead to unnecessary complexity.
  • Initialization Issues: The pattern might require resetting an object's state after cloning, which can be error-prone.

Adapter Pattern

Structural Patterns
import React from 'react';

const TargetButton = ({ label, onClick }) => {
  return <button onClick={onClick}>{label}</button>;
};

const adaptBootstrapButton = (BootstrapButton) => {
  return ({ label, onClick }) => {
    return <BootstrapButton text={label} handleClick={onClick} />;
  };
};

const BootstrapButton = ({ text, handleClick }) => {
  return <button className="btn btn-primary" onClick={handleClick}>{text}</button>;
};

const AdaptedBootstrapButton = adaptBootstrapButton(BootstrapButton);

const App = () => {
  const handleClick = () => {
    alert('Button clicked!');
  };

  return (
    <div>
      <h1>Adapter Pattern Example</h1>
      <TargetButton label="Standard Button" onClick={handleClick} />
      <AdaptedBootstrapButton label="Adapted Bootstrap Button" onClick={handleClick} />
    </div>
  );
};

export default App;

Pros

  • Compatibility: Allows incompatible interfaces to work together, integrating new components with existing systems.
  • Reusability: Enables the reuse of existing code or components that might otherwise be incompatible.
  • Flexibility: Provides a way to use third-party libraries or components without modifying their code.
  • Separation of Concerns: Keeps the core functionality separate from the interface adaptation logic.

Cons

  • Complexity: Introduces additional classes and interfaces, potentially making the overall system more complex.
  • Performance Overhead: The adaptation process may introduce a slight performance cost due to the extra layer of indirection.
  • Maintenance: As the adapted interface evolves, the adapter may need to be updated, requiring ongoing maintenance.
  • Overuse: Excessive use of adapters can lead to a confusing and hard-to-maintain codebase.

Bridge Pattern

Structural Patterns
import React from 'react';

const Abstraction = ({ implementation, title }) => {
  return (
    <div>
      <h2>{title}</h2>
      {implementation.render()}
    </div>
  );
};

const Implementation = {
  render: () => {},
};

const MaterialButton = () => {
  return <button style={{ backgroundColor: 'blue', color: 'white' }}>Material Button</button>;
};

const BootstrapButton = () => {
  return <button className="btn btn-primary">Bootstrap Button</button>;
};

const createImplementation = (type) => {
  switch (type) {
    case 'Material':
      return {
        render: () => <MaterialButton />,
      };
    case 'Bootstrap':
      return {
        render: () => <BootstrapButton />,
      };
    default:
      throw new Error('Unknown implementation type');
  }
};

const App = () => {
  const materialImplementation = createImplementation('Material');
  const bootstrapImplementation = createImplementation('Bootstrap');

  return (
    <div>
      <h1>Bridge Pattern Example</h1>
      <Abstraction implementation={materialImplementation} title="Material UI" />
      <Abstraction implementation={bootstrapImplementation} title="Bootstrap UI" />
    </div>
  );
};

export default App;

Pros

  • Separation of Concerns: Decouples abstraction from implementation, allowing them to vary independently.
  • Flexibility: Enables changing implementations at runtime without affecting the abstraction.
  • Extensibility: New abstractions and implementations can be added easily without modifying existing code.
  • Platform Independence: Useful for cross-platform development where different implementations are needed.

Cons

  • Increased Complexity: Introduces additional interfaces and classes, which can complicate the overall design.
  • Overhead: May result in increased number of classes, potentially impacting performance slightly.
  • Conceptual Difficulty: Can be challenging to understand and implement correctly for developers unfamiliar with the pattern.
  • Potential Overuse: Not every abstraction-implementation separation requires a bridge, and overuse can lead to unnecessary complexity.

Composite Pattern

Structural Pattern
import React from 'react';

const Component = ({ render }) => {
  return render();
};

const Leaf = ({ name }) => {
  return <div>{name}</div>;
};

const Composite = ({ name, children }) => {
  return (
    <div>
      <strong>{name}</strong>
      <div style={{ marginLeft: '20px' }}>
        {children.map((child, index) => (
          <Component key={index} render={child.render} />
        ))}
      </div>
    </div>
  );
};

const App = () => {
  const leaf1 = { render: () => <Leaf name="Leaf 1" /> };
  const leaf2 = { render: () => <Leaf name="Leaf 2" /> };
  const leaf3 = { render: () => <Leaf name="Leaf 3" /> };

  const composite1 = {
    render: () => (
      <Composite name="Composite 1" children={[leaf1, leaf2]} />
    ),
  };

  const composite2 = {
    render: () => (
      <Composite name="Composite 2" children={[composite1, leaf3]} />
    ),
  };

  return (
    <div>
      <h1>Composite Pattern Example</h1>
      <Component render={composite2.render} />
    </div>
  );
};

export default App;

Pros

  • Unified Treatment: Allows individual objects and compositions to be treated uniformly, simplifying client code.
  • Hierarchical Structures: Ideal for representing part-whole hierarchies of objects.
  • Flexibility: Makes it easy to add new types of components without changing existing code.
  • Recursion: Naturally supports recursive operations on the entire tree structure.

Cons

  • Overgeneralization: Can make the design overly general, making it harder to restrict components in compositions.
  • Performance: Operations might be inefficient for deeply nested structures.
  • Complexity: Can complicate the design when dealing with many different types of leaf and composite components.
  • Type Safety: In some languages, it may be challenging to provide type-safe implementations.

Decorator Pattern

Structural Pattern
import React from 'react';

const Button = ({ label, onClick }) => {
  return <button onClick={onClick}>{label}</button>;
};

const withLogging = (Component) => {
  return ({ label, onClick }) => {
    const handleClick = () => {
      console.log(`Button clicked: ${label}`);
      onClick();
    };

    return <Component label={label} onClick={handleClick} />;
  };
};

const withStyle = (Component) => {
  return ({ label, onClick }) => {
    const style = {
      backgroundColor: 'blue',
      color: 'white',
      padding: '10px',
      border: 'none',
      borderRadius: '5px',
    };

    return <Component label={label} onClick={onClick} style={style} />;
  };
};

const App = () => {
  const handleClick = () => {
    alert('Button clicked!');
  };

  const StyledButton = withStyle(Button);
  const LoggingButton = withLogging(Button);
  const StyledLoggingButton = withLogging(withStyle(Button));

  return (
    <div>
      <h1>Decorator Pattern Example</h1>
      <Button label="Simple Button" onClick={handleClick} />
      <StyledButton label="Styled Button" onClick={handleClick} />
      <LoggingButton label="Logging Button" onClick={handleClick} />
      <StyledLoggingButton label="Styled Logging Button" onClick={handleClick} />
    </div>
  );
};

export default App;

Pros

  • Flexibility: Allows dynamic addition of functionalities to objects without altering their structure.
  • Open/Closed Principle: Supports adding new decorators without modifying existing code.
  • Single Responsibility Principle: Each decorator focuses on a specific functionality.
  • Composition over Inheritance: Provides a more flexible alternative to subclassing for extending functionality.

Cons

  • Complexity: Can lead to many small classes, making the system harder to understand and maintain.
  • Order Dependency: The order of applying decorators can affect the final behavior.
  • Overuse: Excessive use can result in code that's difficult to debug and understand.
  • Identity Issues: Decorated components aren't identical to the original component, which can cause issues with reference equality.

Facade Pattern

Structural Pattern
import React from 'react';

const Header = ({ title }) => {
  return <header><h1>{title}</h1></header>;
};

const Content = ({ text }) => {
  return <main><p>{text}</p></main>;
};

const Footer = ({ footerText }) => {
  return <footer><p>{footerText}</p></footer>;
};

const PageFacade = ({ title, text, footerText }) => {
  return (
    <div>
      <Header title={title} />
      <Content text={text} />
      <Footer footerText={footerText} />
    </div>
  );
};

const App = () => {
  return (
    <div>
      <h1>Facade Pattern Example</h1>
      <PageFacade 
        title="Welcome to My Page" 
        text="This is the content of the page." 
        footerText="© 2023 My Website" 
      />
    </div>
  );
};

export default App;

Pros

  • Simplification: Provides a simple interface to a complex subsystem, making it easier to use.
  • Decoupling: Reduces dependencies between clients and subsystems, promoting loose coupling.
  • Layering: Helps in defining entry points to each level of a subsystem, useful in layered architectures.
  • Encapsulation: Hides the complexities of the subsystem from clients, improving maintainability.

Cons

  • Potential for God Object: The facade might become too large and complex if not designed carefully.
  • Limited Flexibility: Clients might need access to lower-level functionality that the facade doesn't expose.
  • Performance Overhead: Can introduce a slight performance cost due to the additional layer of abstraction.
  • Maintenance Challenges: Changes in the subsystem might require updates to the facade, potentially affecting multiple clients.

Flyweights Pattern

Structural Pattern
import React from 'react';

const flyweightFactory = (() => {
  const flyweights = {};

  const getFlyweight = (sharedState) => {
    const key = JSON.stringify(sharedState);
    if (!flyweights[key]) {
      flyweights[key] = sharedState;
    }
    return flyweights[key];
  };

  return {
    getFlyweight,
  };
})();

const FlyweightComponent = ({ sharedState, uniqueState }) => {
  const flyweight = flyweightFactory.getFlyweight(sharedState);
  return (
    <div>
      <p>Shared State: {JSON.stringify(flyweight)}</p>
      <p>Unique State: {uniqueState}</p>
    </div>
  );
};

const App = () => {
  const sharedState1 = { color: 'red', fontSize: '14px' };
  const sharedState2 = { color: 'blue', fontSize: '16px' };

  return (
    <div>
      <h1>Flyweight Pattern Example</h1>
      <FlyweightComponent sharedState={sharedState1} uniqueState="Item 1" />
      <FlyweightComponent sharedState={sharedState1} uniqueState="Item 2" />
      <FlyweightComponent sharedState={sharedState2} uniqueState="Item 3" />
      <FlyweightComponent sharedState={sharedState2} uniqueState="Item 4" />
    </div>
  );
};

export default App;

Pros

  • Memory Efficiency: Shares common data across multiple objects, reducing memory usage.
  • Performance: Can significantly improve performance in systems with a large number of similar objects.
  • Reduced Object Creation: Minimizes the need for creating new objects, which can be resource-intensive.
  • Consistency: Ensures that shared data remains consistent across all instances.

Cons

  • Complexity: Can make the code more complex and harder to understand, especially for developers unfamiliar with the pattern.
  • Debugging Challenges: Shared state can make debugging more difficult, as changes in one place affect multiple objects.
  • Immutability Issues: Can be challenging to implement with immutable objects, which are often preferred in functional programming.
  • Context Sensitivity: The benefits of the Flyweight pattern are highly dependent on how and where it's used in the application.

Proxy Pattern

Structural Pattern
import React from 'react';

const DataComponent = ({ data }) => {
  return (
    <div>
      <h2>Data:</h2>
      <p>User ID: {data.userId}</p>
      <p>ID: {data.id}</p>
      <p>Title: {data.title}</p>
      <p>Body: {data.body}</p>
    </div>
  );
};

const withDataFetching = (Component) => {
  return ({ url }) => {
    const [data, setData] = React.useState(null);
    const [loading, setLoading] = React.useState(true);
    const [error, setError] = React.useState(null);

    React.useEffect(() => {
      const fetchData = async () => {
        try {
          const response = await fetch(url);
          const result = await response.json();
          setData(result);
        } catch (err) {
          setError(err);
        } finally {
          setLoading(false);
        }
      };

      fetchData();
    }, [url]);

    if (loading) return <div>Loading...</div>;
    if (error) return <div>Error: {error.message}</div>;

    return <Component data={data} />;
  };
};

const App = () => {
  const DataComponentWithFetching = withDataFetching(DataComponent);

  return (
    <div>
      <h1>Proxy Pattern Example</h1>
      <DataComponentWithFetching url="https://jsonplaceholder.typicode.com/posts/1" />
    </div>
  );
};

export default App;

Pros

  • Access Control: Allows for controlling and managing access to the main object.
  • Lazy Loading: Can implement lazy initialization and loading of resource-heavy objects.
  • Logging and Caching: Provides a way to add logging and caching mechanisms without changing the original object.
  • Remote Representative: Can act as a local representative for an object in a different address space.

Cons

  • Added Complexity: Introduces an additional layer, which can increase the complexity of the system.
  • Performance Impact: May cause a slight decrease in performance due to the extra level of indirection.
  • Potential for Misuse: If not implemented carefully, it might hide or change the behavior of the original object unintentionally.
  • Synchronization Challenges: In multi-threaded scenarios, ensuring synchronization between the proxy and the real object can be challenging.

Chain of Responsibility Pattern

Behavioral Patterns
import React from 'react';

const createHandler = (handleRequest) => {
  return {
    setNext: function (nextHandler) {
      this.nextHandler = nextHandler;
      return nextHandler;
    },
    handle: function (request) {
      handleRequest(request, this.nextHandler);
    },
  };
};

const logHandler = createHandler((request, next) => {
  console.log('Log Handler:', request);
  if (next) next.handle(request);
});

const authHandler = createHandler((request, next) => {
  if (request.authenticated) {
    console.log('Auth Handler: User authenticated');
    if (next) next.handle(request);
  } else {
    console.log('Auth Handler: User not authenticated');
  }
});

const dataHandler = createHandler((request, next) => {
  console.log('Data Handler:', request.data);
  if (next) next.handle(request);
});

logHandler.setNext(authHandler).setNext(dataHandler);

const App = () => {
  const handleRequest = (request) => {
    logHandler.handle(request);
  };

  const request = {
    authenticated: true,
    data: 'Some important data',
  };

  return (
    <div>
      <h1>Chain of Responsibility Pattern Example</h1>
      <button onClick={() => handleRequest(request)}>Process Request</button>
    </div>
  );
};

export default App;

Pros

  • Flexibility: Allows for dynamic handling of requests through a chain of handlers.
  • Loose Coupling: Decouples senders and receivers of requests, promoting modularity.
  • Simplifies Object Relationships: Reduces dependencies between objects in a system.
  • Extensibility: Easy to add new handlers to the chain without modifying existing code.

Cons

  • Potential Performance Issues: Long chains can lead to slower request processing.
  • Lack of Guaranteed Handling: A request might go unhandled if no appropriate handler is found in the chain.
  • Debugging Challenges: Can be difficult to debug and track the flow of a request through the chain.
  • Potential for Circular References: If not implemented carefully, it can lead to circular references in the chain.

Command Pattern

Behavioral Patterns

type Command = () => void;


const turnOnLight: Command = () => {
  console.log("The light is on");
};

const turnOffLight: Command = () => {
  console.log("The light is off");
};


const Invoker = ({ executeCommand }: { executeCommand: Command }) => {
  return (
    <button onClick={executeCommand}>
      Execute Command
    </button>
  );
};


const App = () => {
  return (
    <div>
      <h1>Command Pattern Example</h1>
      <Invoker executeCommand={turnOnLight} />
      <Invoker executeCommand={turnOffLight} />
    </div>
  );
};

export default App;

Pros

  • Decoupling: Separates the request for an action from its execution, allowing for more flexible systems.
  • Extensibility: New commands can be added without changing existing code, adhering to the Open/Closed principle.

const createBarrier = (numTasks) => {
  let count = 0;
  const waiting = [];

  return {
    await: async () => {
      count++;

    setLog((prevLog) => [...prevLog, `${taskName} reached the barrier`]);
    await barrier.await();
    setLog((prevLog) => [...prevLog, `${taskName} passed the barrier`]);
  };
};

const App = () => {
  const [log, setLog] = useState([]);
  const barrier = createBarrier(3);

  const handleStartTasks = () => {
    const task1 = createTask(barrier, 'Task 1', log, setLog);
    const task2 = createTask(barrier, 'Task 2', log, setLog);
    const task3 = createTask(barrier, 'Task 3', log, setLog);

    task1();
    task2();
    task3();
  };

  return (
    <div>
      <h1>Barrier Pattern Example</h1>
      <button onClick={handleStartTasks}>Start Tasks</button>
      <div>
        <h2>Log:</h2>
        <ul>
          {log.map((entry, index) => (
            <li key={index}>{entry}</li>
          ))}
        </ul>
      </div>
    </div>
  );
};

export default App;

Pros

  • Synchronization: Ensures that a group of related tasks are processed up to a certain point together.
  • Coordination: Useful for dividing a problem into synchronized steps among multiple threads.
  • Load balancing: Can help in achieving better load distribution in parallel processing scenarios.
  • Phased execution: Enables the implementation of algorithms that work in phases, where each phase depends on the completion of the previous one.

Cons

  • Potential for deadlocks: If not all threads reach the barrier, it can lead to a deadlock situation.
  • Performance impact: Threads may need to wait at the barrier, potentially impacting overall performance.
  • Complexity: Adds complexity to the code and can make it harder to reason about the program's flow.
  • Inflexibility: The number of threads must be known in advance, which may not be suitable for all scenarios.

Monitor Pattern

import React, { useState } from 'react';

const createMonitor = () => {
  let isLocked = false;
  const waiting = [];

  return {
    enter: async () => {
      if (isLocked) {
        await new Promise((resolve) => waiting.push(resolve));
      }
      isLocked = true;
    },
    exit: () => {
      if (waiting.length > 0) {
        const resolve = waiting.shift();
        resolve();
      } else {
        isLocked = false;
      }
    },
  };
};

const createCriticalSection = (monitor) => {
  return async (task) => {
    await monitor.enter();
    try {
      await task();
    } finally {
      monitor.exit();
    }
  };
};

const App = () => {
  const [log, setLog] = useState([]);
  const monitor = createMonitor();
  const criticalSection = createCriticalSection(monitor);

  const handleTask = async (taskName) => {
    await criticalSection(async () => {
      setLog((prevLog) => [...prevLog, `${taskName} started`]);
      await new Promise((resolve) => setTimeout(resolve, 1000));
      setLog((prevLog) => [...prevLog, `${taskName} finished`]);
    });
  };

  return (
    <div>
      <h1>Monitor Pattern Example</h1>
      <button onClick={() => handleTask('Task 1')}>Start Task 1</button>
      <button onClick={() => handleTask('Task 2')}>Start Task 2</button>

  );
};

  • Encapsulation: Combines data and the methods that operate on it into a single unit, improving modularity.
  • Mutual exclusion: Ensures that only one thread can access the shared data at a time, preventing race conditions.
